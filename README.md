## 概述

该技能提供了一个结构化的工作流程，用于将 Figma 设计转换为具有像素完美精度的生产就绪代码。它确保与 Figma MCP 服务器的一致集成、设计令牌的正确使用以及与设计 1:1 的视觉对等。

---

## 配置信息

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| 程序代码目录 | 生成的代码存放位置 | `$work_dir/src` |
| Figma 设计数据 | 缓存的 Figma 元数据和图片 | `$work_dir/.figma` |

### 目录结构

```
$work_dir/
├── src/                    # 生成的程序代码
├── .figma/
│   └── $node-id/
│       ├── meta.json       # Figma 节点元数据
│       └── images/         # 下载的图片资源
```

## 环境检查

### 先决条件

以下条件必须满足，否则应暂停并向用户提示错误和解决方法：

1. **figma-developer-mcp MCP Server 必须已链接且可访问**
   - 检查 MCP server 是否可以正常访问
   - 如无法访问，提示用户安装 [Figma-Context-MCP](https://github.com/GLips/Figma-Context-MCP)

2. **用户提供符合格式的 Figma URL**
   - 格式：`https://figma.com/design/:fileKey/:fileName?node-id=1-2`
   - `:fileKey`：Figma 文件密钥
   - `:fileName`：Figma 文件名
   - `:node-id`：要实现的组件或页面的节点 ID
   - 如用户未提供 URL，暂停并要求用户提供

### 补充条件

以下条件非必须，但建议获取以便更好地实现：

1. **技术栈**：用户需提供实现 UI 的技术栈（如未提供，默认使用 React + Tailwind）
2. **代码生成目录**：如未指定，默认在当前目录生成代码

## 工作流程

> **注意**：这是获取 Figma 数据的标准流程，需要严格执行，每次提交新的 URL 都需要重新执行全流程。

### Step 0: 整合 Figma 设计数据

**整合流程（不要跳过）**：

1. **获取 Figma 数据**
   - 运行 `figma_framelink_get_figma_data` 工具
   - 尽可能多地获取 Figma 设计数据
   - 创建 `$work_dir/.figma/$node-id/meta.json` 文件
   - 按照新提交的节点 ID 创建新文件夹
   - 如果数据因过大被截断，可以按深度遍历获取，但需保证 meta.json 的完整性

2. **下载图片资源**
   - 运行 `figma_framelink_download_figma_images` 工具
   - 获取当前 node-id 对应的所有图片资源
   - 保存到 `$work_dir/.figma/$node-id/images` 目录下

3. **数据完整性检查**
   - 仅当同时拥有 meta 数据以及所有图片资源时，认为该节点数据完整
   - 如果有无法获取的文件，暂停并要求用户确认是否跳过

4. **数据更新**
   - 当设计数据更新时，需要重新执行 Step 0，并更新 meta.json 文件

### Step 代码

在1: 实施实施代码前，请检查是否有其他 skill 可以帮助程序实现。

1. **代码生成**
   - 根据 meta.json 文件以及图片资源作为设计和行为的表示
   - 使用用户指定的技术栈生成对应的程序代码
   - 首先检查是否已经有项目代码，如果有，先阅读项目代码，加入到对应位置实现功能
   - 如果没有，程序默认生成在 `$work_dir/src` 目录下
   - 每一个 `$node-id` 对应一个页面，注意模块化设计

2. **组件复用**
   - 注意更多重用现有组件（按钮、输入、排版、图标包装等）而不是重复功能
   - 当存在匹配的组件时，扩展它而不是创建新组件

3. **设计系统一致性**
   - 一致地使用项目的颜色系统、版式比例和间距标记

4. **项目规范**
   - 尊重现有的路由、状态管理和数据获取模式

5. **Auto Layout 配置**
   - 设置内边距（上、右、下、左）
   - 设置项目间距
   - 设置主轴对齐方式（紧凑排列 packed、两端对齐 space-between）

6. **使用变体（Variant）**
   - 定义尺寸：sm、md、lg
   - 定义状态：default、hover、disabled
   - 定义类型：primary、secondary

7. **字阶系统（Type Scale）**
   - 使用 Figma 文本样式定义一套字阶系统
   - 保持字号、字重和行高比例一致
   - 映射为语义化名称：heading-lg、body-md、caption

8. **构建交互原型**
   - 在组件变体之间使用 Smart Animate（智能动画）过渡

9. **自动化任务**
   - 使用 Figma Plugin API 自动化重复性任务
   - 例如批量重命名图层、生成配色方案、导出设计令牌

10. **Dev Mode**
    - 利用 Dev Mode 进行设计交付
    - 检查间距、导出资源
    - 直接从设计中复制 CSS / iOS / Android 代码片段

11. **设计系统文件结构**
    - 包含封面页、变更日志页
    - 按组件类别划分的独立页面（按钮、输入框、导航、反馈等）

### Step 1.2: 代码质量

1. **模块化设计**
   - 遵守 SOLID 原则，提升接口的可维护性和可扩展性

2. **避免硬编码**
   - 提取硬编码数值为常量或设计令牌（design tokens）

3. **组件设计**
   - 保持组件具备良好的可组合性与可复用性

4. **注释**
   - 加入必要注释，包括函数、类、变量等，提高可读性和可维护性

5. **布局规范**
   - 不要在 Auto Layout 框架中使用绝对定位，除非该元素确实需要脱离正常布局流
   - 不要创建脱离主组件的独立实例（detached instances），优先使用变体或属性

6. **图层命名**
   - 不要忽略图层的命名与组织，工程师在 Dev Mode 中需要依赖有意义的图层名称

7. **图片优化**
   - 不要在未优化的情况下直接嵌入全分辨率位图图像

### Step 3: 代码检查

1. **设计一致性**
   - 检查实现代码是否与 Figma 设计一致，要求严格的视觉 1:1 实现

2. **技术栈规范**
   - 检查代码是否符合技术栈的规范

3. **代码质量**
   - 检查代码是否符合项目的代码质量

## 示例

### Example 1: 构建仪表板布局

用户说："实现这个 Figma 设计中的仪表板布局，地址在 https://figma.com/design/:fileKey/:fileName?node-id=1-2"

**Action:**

1. 解析 URL，获取 fileKey、fileName、node-id
2. 检查是否创建了对应的目录结构以及设计文件
3. 运行 `figma_framelink_get_figma_data` 工具，获取 Figma 设计数据
   - 如果已经有 meta.json 文件，对比是否需要更新
   - 如果需要更新，更新 meta.json 文件，否则直接写入文件
4. 运行 `figma_framelink_download_figma_images` 工具，获取图片资源
5. 检查是否已经获取了所有图片资源，如果没有再次请求下载，直到所有图片资源都获取到
6. 检查现有系统是否已经实现了该功能，如果没有，使用项目规范实现该功能
7. 检查实现代码是否与 Figma 设计一致，要求严格的视觉 1:1 实现

**重点检查：**
- 功能是否实现，是否与其他 page 有交互功能
- 组件样式、布局、交互等是否与 Figma 一致
- 代码是否符合项目的代码质量

**最终结果：** 仪表板布局匹配 Figma 设计，与项目设计系统集成。

## 常见问题

### Q1: figma-developer-mcp MCP Server 无法访问

**解决方法：**
- 安装 [Figma-Context-MCP](https://github.com/GLips/Figma-Context-MCP)
- 确保 MCP server 已正确配置并启动

### Q2: 用户未提供 Figma URL

**解决方法：**
- 暂停并要求用户提供符合格式的 Figma URL
- 格式：`https://figma.com/design/:fileKey/:fileName?node-id=1-2`

### Q3: Figma 数据获取被截断

**解决方法：**
- 使用 depth 参数按深度遍历获取数据
- 确保最终 meta.json 的完整性

### Q4: 图片资源下载不完整

**解决方法：**
- 多次调用下载工具直到所有图片资源都获取到
- 如果仍有无法获取的文件，暂停并要求用户确认是否跳过

### Q5: 设计数据更新后如何处理

**解决方法：**
- 重新执行 Step 0，更新 meta.json 文件
- 重新下载图片资源
- 重新执行 Step 1 和 Step 3

### Q6: 代码与设计不一致

**解决方法：**
- 严格按照 meta.json 中的布局、样式、间距等信息实现
- 使用 Dev Mode 检查设计细节
- 确保像素级别的视觉对等
